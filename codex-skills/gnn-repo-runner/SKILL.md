# GNN 仓库执行器

## 概述

把一个原始 GNN 仓库变成可运行、可追踪、可复用的执行流程。
强调执行纪律：先扫描，再运行最小可用命令，修复阻塞问题，并保留完整日志。

## 工作流

1. 立即创建运行日志。
在类似 `reproduction_notes/<repo-name>-run.md` 的路径创建或更新 Markdown 日志，把所有关键动作集中记录：
- 仓库扫描结果
- 环境准备过程
- 执行过的命令
- 报错与修复
- 阶段性结果
- 下一步动作

2. 运行前先扫描。
检查：
- README 和安装说明
- `requirements.txt`、`environment.yml`、`pyproject.toml` 等依赖文件
- 训练和评测入口
- 配置文件和 shell 脚本
- 数据集要求
- 现有 checkpoint、日志和输出目录

3. 选择最小但有意义的第一次运行。
优先顺序通常是：
- 带 `--help` 的命令
- 基于已有 checkpoint 的纯评测命令
- 短训练 smoke test
- 单数据集、单 seed 的主训练命令

4. 用最小改动修复阻塞问题。
常见阻塞包括：
- 依赖缺失
- 路径假设不成立
- CUDA 或设备不匹配
- PyTorch、PyG、DGL、NumPy 的 API 演化
- 硬编码文件路径
- shell 脚本和本地平台不兼容

5. 记录每一个关键动作。
不要等到最后再回忆。命令、目的、结果、文件改动都要边做边写进日志。

6. 区分“能运行”和“能复现”。
仓库跑起来之后，要明确它目前只是可执行，还是已经能和论文结果做科学对比。如果还不能对比，要指出还缺哪些环节。

## 工作规则

- 优先使用原始入口，不要上来就自己包新启动器。
- 代码改动应尽量少、可回退、理由充分。
- 明确区分环境修复和科学改动。
- 如果依赖装不上或数据集拿不到，要精确说明 blocker，并同步更新日志。
- 与其留下很多半成品尝试，不如优先得到一次干净的成功运行。
- 默认使用中文回复用户，默认使用中文撰写运行日志、实验总结和 Markdown 文档；仅在代码、命令、路径、报错原文必须保留英文时使用英文。

## 交付内容

完成一次仓库执行任务时，至少输出：
- 运行日志路径
- 识别到的主要入口
- 真正跑通的命令
- 必要的代码或配置修复
- 当前执行状态：未跑通、部分跑通或完全跑通
- 达到论文级复现还需要的下一步

## 资源

- 使用 `scripts/init_repo_run_log.py` 在动仓库前先生成运行日志模板。
- 当仓库结构混乱、出现跨平台问题，或不清楚先跑什么时，阅读 `references/runbook.md`。

## 示例请求

- "Use $gnn-repo-runner，检查这个原始 GNN 仓库，跑通主实验，并记录每一个关键步骤。"
- "Use $gnn-repo-runner，把这个代码库在我的机器上跑起来，修掉阻塞问题，并留下一份清晰的运行日志。"
