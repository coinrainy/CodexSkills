# GNN 实验复盘

## 概述

把原始日志、结果表、曲线和零散 ablation 转成清晰诊断和具体下一步实验计划。
重点是找出信息增益最高的下一组实验，而不是写很长的叙述。

## 工作流

1. 重建到底改了什么。
列出 baseline 和新实验之间的精确差异：seed 数量、数据划分、增强强度、模型结构、优化器、训练日程和评测代码。

2. 先描述症状，再讨论原因。
常见症状包括：
- 平均值提升但方差很大
- 跨 seed 稳定退化
- train 提升但 validation 卡住
- 新正则项或新视图加入后训练崩掉
- loss 不稳定或梯度爆炸
- 只在一个数据集上有效、其他数据集失效

3. 给出相互竞争的假设。
当证据不足时，至少给出两个合理解释。每个解释都标明更像是代码问题、科学问题，还是暂时无法判定。

4. 为信息增益设计下一批实验。
优先让每个实验只隔离一个因素。好的下一批实验通常包括：
- 一个确认性运行
- 一个去掉疑似有效组件的 ablation
- 一个更难或对照性质更强的数据集压力测试
- 如果怀疑是实现问题，再加一个廉价诊断实验

## 工作规则

- 结论要说明它来自原始日志、聚合指标，还是推理判断。
- 没有方差背景时，不要把很小的提升说成有意义。
- 如果 seed 数不够，下一步优先建议补 seed，而不是继续想新 idea。
- 当“best checkpoint”和“final epoch”有区别时，要明确区分。
- 当一个改动变差时，要解释更像是机制不匹配、调参不当，还是实现问题。
- 默认使用中文回复用户，默认使用中文撰写分析、复盘和 Markdown 文档；代码、命令、路径和必要术语保留英文。

## 交付内容

完成一次实验复盘时，至少输出：
- 一段总体判断
- 主要症状列表
- 带置信度排序的假设
- 按优先级排列的下一批实验
- 哪个结果会推翻当前最强假设的简短说明

## 资源

- 使用 `scripts/init_review_note.py` 为一批运行结果生成结构化复盘笔记。
- 当失败模式不明确，或多个症状相互影响时，阅读 `references/diagnosis.md`。

## 示例请求

- "Use $gnn-experiment-review，分析我最近这批 GNN 实验，并告诉我接下来最该做的 3 个实验。"
- "Use $gnn-experiment-review，解释为什么我的新增强在一个数据集上变好、在其他数据集上变差。"
